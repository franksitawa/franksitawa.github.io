Manim Voiceover ships with built-in support for services like OpenAI and Azure, but if you are running XTTS locally in Docker, none of those apply. What you need is a custom service that talks directly to your local XTTS API.

This post walks through building one. If you have not yet set up XTTS or Manim Voiceover, start with these two posts first:

- [Running XTTS-v2 with Docker and NVIDIA CUDA on Ubuntu](https://franksitawa.github.io/ai/tts/docker/2026/06/25/xxts.html)

## How the Plugin Works

Manim Voiceover expects a speech service that extends `SpeechService` and implements a single method: `generate_from_text`. When your scene hits a `with self.voiceover(...)` block, the plugin calls this method, receives a path to the generated audio file, measures the duration, and exposes it as `tracker.duration` so your animation can sync to it.

That contract is all you need to implement.

---

## The Custom Service

Create a file named `xtts_service.py` in your project directory:

```python
import requests
from manim_voiceover.services.base import SpeechService


class XTTSService(SpeechService):

    def __init__(
        self,
        api_url="http://localhost:8020/tts_to_audio",
        speaker_wav="speaker.wav",
        language="en"
    ):
        super().__init__()
        self.api_url = api_url
        self.speaker_wav = speaker_wav
        self.language = language

    def generate_from_text(
        self,
        text,
        cache_dir=None,
        path=None,
        **kwargs
    ):
        payload = {
            "text": text,
            "speaker_wav": self.speaker_wav,
            "language": self.language
        }

        response = requests.post(self.api_url, json=payload)
        response.raise_for_status()

        with open(path, "wb") as f:
            f.write(response.content)

        return {
            "original_text": text,
            "input_text": text,
            "output_file": path
        }
```

The implementation is intentionally minimal. It sends the text and speaker configuration to XTTS, writes the returned audio bytes to `path`, and returns the metadata dictionary Manim Voiceover expects.

> **Note:** The exact API endpoint may differ depending on which XTTS server image you are running. The `daswer123/xtts-api-server` image used in the Docker setup post exposes `/tts_to_audio` by default. Check your container's docs at `http://localhost:8020/docs` if you get a 404.

## Using the Service in a Scene

With `xtts_service.py` in the same directory as your scene file, you can import and use it directly:

```python
from manim import *
from manim_voiceover import VoiceoverScene
from xtts_service import XTTSService


class BinarySearchVideo(VoiceoverScene):

    def construct(self):
        self.set_speech_service(XTTSService())

        title = Text("Binary Search")

        with self.voiceover(
            text="Binary search repeatedly cuts the search space in half."
        ) as tracker:
            self.play(
                Write(title),
                run_time=tracker.duration
            )
```

`tracker.duration` is derived from the actual length of the audio clip XTTS generates. This means the animation timing adjusts automatically to match the narration — no hardcoded `run_time` values, no manual syncing.

## Why Not Just Use a Cloud Service?

The built-in OpenAI and Azure services work, but they introduce dependencies that matter at scale:

- **Cost** — every render call bills against your API quota. When you are iterating on a scene and re-rendering dozens of times, this adds up fast.
- **Latency** — cloud round-trips slow down your development loop.
- **Voice consistency** — cloud TTS voices are fixed. With XTTS you clone a specific voice once and reuse it across every video in your series, giving your content a consistent identity.
- **Offline capability** — once the Docker container is running, the pipeline works without an internet connection.

For a one-person content production setup, eliminating API costs and keeping everything local makes the workflow significantly more sustainable.

## Extending the Service

A few useful additions once you have the basics working:

**Switch voices per scene** — pass a different `speaker_wav` when instantiating `XTTSService()` to use different voices for different types of content.

**Add caching** — XTTS generation takes a moment. If you render the same voiceover line repeatedly during development, caching the audio file by a hash of the text string avoids redundant API calls.

**Plug into a pipeline** — because the service is plain Python, it can be called from n8n, LangGraph, or any automation layer. You can build a workflow where lesson scripts are generated by an LLM, passed to Manim scenes, and rendered to video with narration — all without manual intervention.

## The Bigger Picture

This custom service is the bridge between the XTTS Docker setup and Manim's animation pipeline. Once it is in place, the full workflow from text to narrated video runs entirely from Python, entirely on your local machine:

1. Write the lesson script
2. Define the Manim scene
3. Render — XTTS generates audio, Manim syncs and composites
4. FFmpeg handles final output formatting

That is the foundation I use for producing animated CS explainers, and it scales cleanly as the content library grows.
