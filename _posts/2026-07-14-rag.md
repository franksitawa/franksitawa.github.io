Large language models are confident by default — including when they're wrong. Ask one about something outside its training data, your private notes, or last week's news, and it will often answer anyway, fluently and incorrectly. Retrieval-Augmented Generation (RAG) is the standard fix: instead of trusting the model's memory, you hand it the relevant facts at answer time and ask it to work from those.

This post covers the theory in brief, then walks through running and understanding a minimal but complete RAG implementation — [`rag-demo`](https://github.com/franksitawa/rag-demo), an Express + React app wired to the free Gemini API. It's the same project I used in a recent session at [Eldohub Eldoret](https://eldohub.co.ke) on 3rd July 2026 — [photos from the event here](https://eldohubphotography.pixieset.com/fridaycombinedclass/) — and this write-up is the companion reference for anyone who wants to rebuild it step by step.

## Why RAG?

Before the code, it's worth being clear about what problem RAG actually solves. Three gaps show up whenever you rely on a model's memory alone:

- **Knowledge cutoff** — training data has a fixed end date. Anything that happened after that date simply doesn't exist for the model.
- **No access to private data** — a model can't answer questions about your own notes, docs, or codebase, because it was never trained on them.
- **Hallucination** — when a model is uncertain, it tends to generate a plausible-sounding answer rather than admit it doesn't know.

RAG addresses all three the same way: retrieve the relevant text from a knowledge source you control, insert it into the prompt as context, and instruct the model to answer *from that context* — and to say so if the context doesn't contain the answer. The model still does the writing; it just stops writing from memory alone.

The mechanism has three steps:

1. **Retrieve** — find the chunks of stored text most relevant to the question.
2. **Augment** — insert those chunks into the prompt as grounding evidence.
3. **Generate** — the model answers using the supplied context.

Steps 1 and 2 are what separate RAG from a plain chatbot, and they're what this demo makes visible.

## How Retrieval Actually Works

Retrieval depends on two ideas: embeddings and vector similarity.

An **embedding** is a numeric representation of text — a long list of floating-point numbers — produced by an embedding model such that texts with similar meaning end up close together in that numeric space, regardless of the exact words used. "Affordable airfare" and "cheap flights" land near each other even though they share no words.

To find relevant chunks for a question, the app embeds the question and compares it against every stored chunk's embedding using **cosine similarity** — the cosine of the angle between two vectors. A smaller angle means a higher similarity score. The top-scoring chunks are the ones handed to the model as context.

A **vector database** is just infrastructure built around this idea: store embeddings alongside their source text, and answer "which vectors are closest to this one?" quickly, even across millions of records, using approximate nearest-neighbor indexes instead of comparing against every vector one by one. This demo skips the index — it's a plain array with a cosine similarity function — because at a few hundred chunks, brute force is fast enough to see the mechanism clearly. The concept is identical to what Pinecone, Chroma, pgvector, or Weaviate do at scale.

## Setting Up the Demo

The full source is at [github.com/franksitawa/rag-demo](https://github.com/franksitawa/rag-demo). It's a two-process app: an Express backend that handles embedding, storage, and retrieval, and a React (Vite) frontend for adding documents and asking questions.

```
rag-demo/
  backend/
    server.js        # Express routes: /api/documents, /api/query
    gemini.js         # Calls Gemini embedContent + generateContent REST endpoints
    vectorStore.js     # In-memory chunking + cosine similarity search
    .env.example
    package.json
  frontend/
    src/App.jsx        # UI: knowledge base panel + query panel + pipeline trace
    src/App.css
    src/index.css
    package.json
    vite.config.js     # Proxies /api to the backend during dev
```

### 1. Get a free Gemini API key

Go to [aistudio.google.com/apikey](https://aistudio.google.com/apikey) and create a key. Google AI Studio's free tier is generous enough to run this demo without paying anything.

### 2. Clone the repo and start the backend

```bash
git clone https://github.com/franksitawa/rag-demo.git
cd rag-demo/backend
npm install
cp .env.example .env
# edit .env and paste your GEMINI_API_KEY
npm start
```

The server runs on `http://localhost:3001`.

### 3. Start the frontend

In a second terminal:

```bash
cd rag-demo/frontend
npm install
npm run dev
```

Open the URL Vite prints — usually `http://localhost:5173`.

### 4. Try it

1. Paste a chunk of text into the **Knowledge base** panel — an article, some notes, anything — and click **Add to knowledge base**. Add a few separate pieces if you like; each is chunked and embedded independently.
2. Type a question related to what you pasted and click **Ask**.
3. Read the answer, then check the retrieved sources underneath — you'll see exactly which chunks were pulled in and their similarity scores.

Ask something the pasted text doesn't cover, too. The prompt instructs Gemini to say it doesn't have enough information rather than guess — that instruction only works because retrieval is doing its job correctly.

## Walking Through the Code

### `gemini.js` — talking to the API

Both embedding and generation go through Gemini's REST API directly, with no SDK dependency. The key detail is `taskType`: chunks being stored use `RETRIEVAL_DOCUMENT`, while the user's question uses `RETRIEVAL_QUERY`. Gemini optimizes the resulting vectors differently depending on which side of the search they're going to be on.

```javascript
export async function embedText(text, taskType = "RETRIEVAL_DOCUMENT") {
  const apiKey = getApiKey();
  const url = `${GEMINI_BASE}/models/${EMBEDDING_MODEL}:embedContent?key=${apiKey}`;

  const res = await fetch(url, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      model: `models/${EMBEDDING_MODEL}`,
      content: { parts: [{ text }] },
      taskType,
    }),
  });

  const data = await res.json();
  return data.embedding.values; // array of floats
}
```

Generation is a separate call — `generateAnswer(prompt)` — that sends a fully-assembled prompt (question plus retrieved context) to `generateContent` and returns the model's text response.

### `vectorStore.js` — the retrieval logic

This file is the entire "vector database" in the demo: chunking, storage, and cosine similarity search, in about sixty lines.

Chunking splits long text into overlapping pieces so an idea that straddles a boundary isn't cut in half:

```javascript
export function chunkText(text, chunkSize = 500, overlap = 80) {
  const cleaned = text.replace(/\s+/g, " ").trim();
  const chunks = [];
  let start = 0;
  while (start < cleaned.length) {
    const end = Math.min(start + chunkSize, cleaned.length);
    chunks.push(cleaned.slice(start, end).trim());
    if (end === cleaned.length) break;
    start = end - overlap;
  }
  return chunks.filter(Boolean);
}
```

Similarity search compares the query vector against every stored vector and returns the highest-scoring matches:

```javascript
function cosineSimilarity(a, b) {
  let dot = 0, normA = 0, normB = 0;
  for (let i = 0; i < a.length; i++) {
    dot += a[i] * b[i];
    normA += a[i] * a[i];
    normB += b[i] * b[i];
  }
  return dot / (Math.sqrt(normA) * Math.sqrt(normB));
}

export function similaritySearch(queryVector, k = 4) {
  const scored = store.map((item) => ({
    ...item,
    score: cosineSimilarity(queryVector, item.vector),
  }));
  scored.sort((a, b) => b.score - a.score);
  return scored.slice(0, k);
}
```

Swap this file for a call to Pinecone or pgvector and nothing else in the app needs to change — that's the point of separating retrieval logic from the rest of the pipeline.

### `server.js` — wiring it together

Two routes do all the work. `POST /api/documents` chunks and embeds pasted text:

```javascript
app.post("/api/documents", async (req, res) => {
  const { text, source } = req.body;
  const chunks = chunkText(text);
  const vectors = await embedBatch(chunks, "RETRIEVAL_DOCUMENT");
  const added = chunks.map((chunkTextValue, i) =>
    addChunk(chunkTextValue, vectors[i], source || "pasted-text")
  );
  res.json({ chunks: added, totalChunksInStore: storeSize() });
});
```

`POST /api/query` is the retrieval-augmentation-generation sequence in one place: embed the question, retrieve the top chunks, build a grounded prompt, and generate the answer.

```javascript
app.post("/api/query", async (req, res) => {
  const { question, k = 4 } = req.body;

  const queryVector = await embedText(question, "RETRIEVAL_QUERY");
  const topChunks = similaritySearch(queryVector, k);

  const context = topChunks
    .map((c, i) => `[Source ${i + 1}] ${c.text}`)
    .join("\n\n");

  const prompt = `Answer using ONLY the context below. If the answer isn't
there, say you don't have enough information.

Context:
${context}

Question: ${question}
Answer:`;

  const answer = await generateAnswer(prompt);
  res.json({ answer, sources: topChunks });
});
```

Everything upstream of this route — chunking, embedding, storage — exists to make this one function call possible: by the time the prompt is built, the model only has to reason over a handful of relevant paragraphs instead of guessing from its training data.

### `App.jsx` — showing the mechanism, not hiding it

The frontend deliberately doesn't hide the pipeline behind a plain chat box. A small trace animates through *ingest → embed → store* and *embed query → retrieve → augment → generate* as each request runs, and every answer is shown next to the exact chunks and similarity scores that produced it. For a demo whose whole purpose is explaining how RAG works, that visibility matters more than a polished chat UI would.

## How This Maps to Production RAG

The demo keeps every piece simple enough to read in one sitting, which means it also cuts corners a production system wouldn't:

| This demo | Production equivalent |
|---|---|
| In-memory array + cosine similarity | Pinecone, Chroma, pgvector, Weaviate, Qdrant, etc. |
| Fixed-size character chunking with overlap | Semantic/structural chunking (by heading, sentence, token count) |
| Chunks re-embedded on every restart, nothing persisted | Persisted vector DB with incremental indexing |
| Single free-tier key, sequential embedding calls | Batched calls, rate limiting, retries |

None of those upgrades change the underlying pattern — retrieve, augment, generate — they just make it hold up at scale.

## Where to Go From Here

If you want to extend the demo, a few directions are natural next steps:

- **Swap the vector store** for pgvector or Chroma so the knowledge base survives a restart.
- **Add metadata filtering** — tag chunks by source or date and filter before running similarity search.
- **Chain it into a pipeline** — feed it documents from n8n or a scheduled job instead of pasting text by hand, the same way the [XTTS voiceover service](/ai/education/manim/2026/06/28/building-a-custom-xtts-service-for-manim-voiceover.html) plugs into an automated video pipeline.

The full code is on GitHub at [franksitawa/rag-demo](https://github.com/franksitawa/rag-demo) — clone it, add your own notes to the knowledge base, and see what it gets right and where it falls back to "I don't have enough information."
